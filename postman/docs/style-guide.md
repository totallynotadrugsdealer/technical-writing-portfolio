# Style Guide

## Purpose

This style guide defines the writing standards used throughout this documentation project. The goal is to produce documentation that is clear, consistent, accurate, and easy for beginners to follow.

---

# Audience

This documentation is written for users with little or no prior experience working with APIs or Postman.

Assume the reader is technically curious but unfamiliar with API terminology.

---

# Writing Principles

- Prioritize clarity over technical complexity.
- Explain unfamiliar terminology before using it.
- Keep instructions concise.
- Write from the user's perspective.
- One topic per document.

---

# Voice and Tone

Voice:
Professional

Tone:
Friendly, encouraging, and instructional.

Avoid humour, slang, and unnecessary jargon.

---

# Point of View

Write in the second person.

✔ You can send a request.

✘ The user can send a request.

---

# Tense

Use present tense whenever possible.

✔ Postman stores requests in Collections.

✘ Postman will store requests in Collections.

---

# Headings

Use sentence case.

✔ Create your first request

✘ CREATE YOUR FIRST REQUEST

---

# Procedures

Use numbered steps.

Each step should contain one action.

---

# User Interface Elements

Use bold formatting for buttons, menu items, and interface labels.

Example:

Select **Send**.

Open **Collections**.

---

# Notes

Use blockquotes.

> Note:
> You must be connected to the internet to sign in.

---

# Screenshots

Include screenshots only when they clarify an action or interface.

Avoid screenshots that contain unnecessary UI elements.

---

# Code

Use fenced code blocks.

Specify the language whenever possible.

---

# Terminology

Use terms consistently.

Always capitalize:

- Postman
- Collection
- Workspace
- Environment

Avoid abbreviations unless previously explained.

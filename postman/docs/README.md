# Postman Documentation Project

## Overview

This project is part of my **Technical Writing Portfolio** and demonstrates my approach to planning, researching, structuring, writing, and maintaining end-user software documentation.

Rather than documenting a fictional product, I chose **Postman**, a widely used API development platform, to create documentation that reflects real-world technical writing practices. The goal of this project is not to replace the official Postman documentation, but to produce a beginner-oriented learning guide that introduces Postman's core concepts through clear, structured, task-based documentation.

This documentation has been written with first-time users in mind and follows a progressive learning path, beginning with foundational concepts before introducing practical tasks and troubleshooting techniques.

---

# Project Goals

This project was created to demonstrate the ability to:

- Research an existing software product.
- Analyse the needs of a beginner audience.
- Plan a complete documentation set before writing.
- Create consistent information architecture.
- Write clear task-based and concept-based documentation.
- Apply a documentation style guide consistently.
- Organise content into logical learning stages.
- Use Markdown and Git for documentation version control.
- Incorporate screenshots to support written instructions.
- Maintain documentation using iterative commits.

# Documentation Process

The documentation was developed using a structured technical writing workflow rather than writing articles independently.

Before writing began, a documentation plan was created to define:

## 2. Product Research

The official Postman website and Learning Center were reviewed to understand:

- Product capabilities
- User workflows
- Terminology

Research findings were documented before writing began.

Deliverable:

```
research-notes.md
```

## 3. Information Architecture

The documentation was organised into a logical learning journey.

Instead of documenting every Postman feature, the documentation was divided into beginner-friendly stages that gradually introduce new concepts.

The final documentation structure is:

```
Stage 1 – Understanding the Product

01 What is Postman?
02 What is an API?
03 Understanding HTTP Requests and Responses

Stage 2 – Getting Started

04 Installing Postman
05 Creating a Postman Account
06 Understanding the Postman Interface

Stage 3 – First API Request

07 Sending Your First GET Request
08 Reading the Response
09 Saving a Request

Stage 4 – Organising Work

10 Collections
11 Workspaces
12 Environments

Stage 5 – Troubleshooting

13 Common Errors
14 Authentication Problems
15 Network Errors

Reference

16 Glossary
17 Keyboard Shortcuts
18 HTTP Status Codes
```

Deliverable:

```
information-architecture.md
```

## 4. Style Guide

A documentation style guide was created before drafting the content to ensure consistency across every document.

The style guide defines:

- Tone
- Voice
- Grammar
- Heading hierarchy
- Capitalisation
- Lists
- Notes and tips
- Image captions
- Terminology
- Formatting conventions

Deliverable:

```
style-guide.md
```

---

## 5. Writing

Each document was written according to the documentation plan and style guide.

Every guide follows a consistent structure, including:

- Purpose
- Prerequisites
- Concept or Task
- Verification
- Summary
- Related documentation

This allows readers to move through the documentation in a predictable and structured manner.

---

## 6. Visual Documentation

Screenshots were captured throughout the project to support written instructions.

Images were:

- Captured during real product usage.
- Cropped where appropriate.
- Named consistently.
- Referenced within Markdown documents.
- Captioned using technical documentation conventions.

Where official Postman diagrams were used, attribution has been provided.

---

## 7. Version Control

The project is maintained using Git and GitHub.

Rather than committing only completed documents, development was recorded through incremental commits that capture the documentation workflow.

This repository currently contains the complete revision history of the project.
